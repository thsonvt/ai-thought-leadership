If I could summon a fleet of <u><a href="https://every.to/guides/agent-native" rel="noopener noreferrer" target="_blank">AI agents</a></u>, let them coordinate, and watch them produce work no single agent could match, it would relieve some of my overwhelm. </p><p>I tried everything to make it work—<u><a href="https://every.to/source-code/how-i-use-claude-code-to-ship-like-a-team-of-five" rel="noopener noreferrer" target="_blank">Claude Code</a></u> teams, agents dispatching tasks to other agents, <u><a href="https://every.to/source-code/the-three-ways-i-work-with-llms" rel="noopener noreferrer" target="_blank">orchestration setups</a></u> where a lead agent managed a pool of workers. Many iterations, many burned tokens.</p><p>But more agents didn’t make me faster. I’ve <u><a href="https://every.to/source-code/how-i-use-claude-code-to-ship-like-a-team-of-five" rel="noopener noreferrer" target="_blank">run parallel Claude Code sessions</a></u> for months, which works when each agent has a clear task, and I’m directing the work. The swarm experiment was different: agents coordinating with each other, deciding what to work on, producing output I hadn’t shaped. When 10 of them finished simultaneously, I had 10 <u><a href="https://every.to/source-code/i-stopped-reading-code-my-code-reviews-got-better" rel="noopener noreferrer" target="_blank">results to evaluate</a></u> without enough context to know which ones I could trust. AI agents don’t have a speed limit, but the person managing them still does.</p><p>I kept looking for a smarter orchestration layer—a better protocol or a tighter framework that would filter the output and tell me which result to trust. Then I stopped and looked at what was really doing the work.</p><p>It was something I already had—a folder.</p><p>A project folder with a CLAUDE.md/AGENT.md (the file that tells an AI how to work in your project), some <u><a href="https://skills.every.to/" rel="noopener noreferrer" target="_blank">skill</a></u> definitions, and context accumulated through months of <u><a href="https://every.to/source-code/compound-engineering-the-definitive-guide" rel="noopener noreferrer" target="_blank">compound engineering</a></u>—that’s an agent. The context that this folder gives an AI model makes the generalized model a specialist in whatever task or field you want it to excel in. </p><p>I’m running 44 of these folders-as-agents across multiple projects now. Each one runs inside a specialized folder I’ve built and tested over months, and a dispatch layer I built on top does the routing between them. Here’s how it works.  </p><h2><strong>The agents hiding on your hard drive</strong></h2><p>People hear “agent” and picture a <u><a href="https://en.wikipedia.org/wiki/Rube_Goldberg_machine" rel="noopener noreferrer" target="_blank">Rube Goldberg machine</a></u>—dozens of comically complex moving parts, each one triggering the next. But an agent is much simpler: a model with enough context so you don’t have to re-explain everything each time you open the chat. </p><p>Here’s an example: All of Cora’s code lives in a project folder in the Every organization on GitHub. When I open that folder with Claude, Claude can see the code and the structure. But it doesn’t know my way of working or what I care about, which is why the folder also includes a <u><a href="http://claude.md" rel="noopener noreferrer" target="_blank">CLAUDE.md</a></u> file. The file tells Claude how I name things and how I structure tests. That’s an agent—not a fancy one, but an agent nonetheless. Just by pointing the model at this folder, which contains some of my personality, knowledge, and <u><a href="https://every.to/p/what-is-taste-really" rel="noopener noreferrer" target="_blank">taste</a></u>, the model can be a specialist in my codebase. </p><p>Claude Skills—files that give the model specific capabilities—are an example of this “folder as agent” structure.
